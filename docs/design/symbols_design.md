# Debug Symbols Design

## Summary
The Debug Symbols data is a read-only JSON artifact generated by the assembler or compiler.
It contains source-level symbol information and is consumed by the emulator and editor to map
compiled addresses back to source code metadata, enabling source-level debugging features.

## Requirements
- JSON file stored next to the ROM/BIN file during the compilation time.
- `*.symbols.json` filename.
- Keep source paths project-relative for portability.
- line is a 1-based source line number
- Symbol type can be `label`, `const`, `func`, `macro`, `macroparam`.
- Data lines describes source lines that emit contiguous data bytes into memory, typically directives like .byte/.db and .word/.dw.
- Data lines address is a starting memory address of the data emitted by that source line.
- Data lines byte length is the total number of bytes emitted by that single source line. Used for boundary check.
- Data lines unit bytes is the size in bytes of one logical element on that line. For .byte it's 1, for .word it's 2, etc.
- A C compiler uses actual function metadata for function symbols.
- An assembler uses `.optional` / `.endoptional` (short: `.opt` / `.endopt`) for function boundary detection.
- Numeric addresses in a range of 0 to 0xFFFF.
- Debug symbols is read-only data.

## Symbol Naming
- Global labels and constants use their source name as-is: `START`, `MAX_SIZE`.
- Local labels (`@name`) are disambiguated by occurrence index within the file: `@loop_0`, `@loop_1`.
- Macro params are keyed as `MacroName.paramName`: `SetPalette.n`.
- Variables (`.var`) share the `const` type.

## Structure
```json
{
  "symbols": {
    "<symbolName>": {
      "value": number,
      "path": string,
      "line": number,
      "type": "label" | "const" | "func" | "macro" | "macroparam"
    }
  },
  "lineAddresses": {
    "<path>": {
      "<line>": [addr, ...]
    }
  },
  "dataLines": {
    "<path>": {
      "<line>": {
        "addr": number,
        "byteLength": number,
        "unitBytes": number
      }
    }
  }
}
```

## Usage
- Symbol's debug metadata (compile time and runtime values) on hover.
- Definition lookup.
- Currently executed source line highlighting.
- Data-line (.byte, .word, etc.) memory mapping, read/write highlight, compile time and runtime values.
- For labels a value is an address.
- For consts a value is used as it is.
- If a macroparam have no default value, value is -1.
- For funcs a value is an address.
- For macros a value is not used. look up all addresses from lineAddresses.


## Testing Strategy
### Unit
- Serialize each symbol type (`label`, `const`, `func`, `macro`, `macroparam`) and assert all JSON fields are correct.
- Local label disambiguation: two `@loop` labels in the same file produce `@loop_0` and `@loop_1`.
- Macro param with default value serializes to the evaluated numeric default; absent default emits `-1`.
- `dataLines` path-grouping: verify `path → line → {addr, byteLength, unitBytes}` structure is preserved.
- `lineAddresses` multi-address: a single line that emits bytes at two addresses records both in the array.
- Multi-file project: symbols from included files appear under their relative paths.
- Symbol name collision: local label from different scopes do not overwrite each other.

### Integration
- Compile sample project and assert required sections exist.
- Symbol hover/definition correctness across included files.

### Regression
- Validate behavior with missing optional sections (`dataLines`).
- Update documentation on every feature change.
